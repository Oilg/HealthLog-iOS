# HealthLog iOS — правила работы

## Ветки

- Перед началом любой задачи проверить наличие активной невмерженной ветки (`git branch -a`, `gh pr list`).
- Если активной ветки нет — создать новую от `master`.
- Одна задача — одна ветка.

## Workflow после выполнения задачи

1. **Ревью** — запустить `/review` на текущей ветке.
2. Если ревью нашло замечания — исправить, затем снова `/review`.
3. Если ревью прошло успешно — запушить ветку и смержить в `master`.
4. После мержа в `master` — проверить GitHub Actions:
   - CI должен пройти все джобы: lint, format, unit tests, build, analyzer, dead code, UI tests.
   - Deploy to TestFlight должен завершиться успешно.

## Если CI падает

- Открыть упавшую джобу: `gh run view <run-id> --log-failed`.
- Исправить ошибки, запушить фикс в ту же ветку.
- Убедиться что CI прошёл повторно.

## Если деплой в TestFlight падает

- Открыть логи: `gh run view <run-id> --log-failed`.
- Исправить ошибки (сертификаты, build settings, Fastlane), запушить.
- Убедиться что деплой прошёл повторно.

## Линтер и форматирование

- SwiftLint и SwiftFormat проверяются в CI — код должен проходить до мержа.
- Строка до 160 символов.
- Имена идентификаторов минимум 2 символа.
- Файл до 400 строк (warning).

## Тесты

- Покрытие unit-тестами ≥ 15% (CI gate).
- После добавления нового кода — добавить тесты.
